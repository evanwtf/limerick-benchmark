## Key Findings

**1. Gemma family went 4/4; Qwen family went 0/6.**
Every gemma4 variant passed (e2b, e4b, e2b-mlx-bf16, 26b). No Qwen model produced a working server — five tripped aider's loop detection, one finished without ever opening the port.

**2. Two distinct Qwen failure modes.**
- **Short log-cycle loop (~60–100 s):** 9b, 35b-a3b, 3.6:35b-a3b-q8_0 all produced a tight repeating log pattern that aider's cycle detector caught before the 180-second stagnation guard would have.
- **Full-window stagnation (180 s):** 27b-coding-mxfp8 and 3.6:latest spent the whole window planning without touching the workspace; the hash-based watchdog aborted them.

**3. One silent failure worth flagging.**
`qwen3.5:35b-a3b-coding-mxfp8` reported `finish_reason: completed` but the evaluator got `port_never_opened`. Aider thinks it shipped; the server disagrees. That's a case the loop-detection guards can't catch — worth keeping an eye on if this pattern recurs.

**4. Run-to-run variance is real.**
Compared to job `20260417_073034` (same 10 models, same aider agent):
- `gemma4:26b` failed before (planning paralysis, 70 s) → **passed now** (52.8 s) with a class-based generator.
- `qwen3.5:9b` passed before (95 s) → **failed now** (log cycle, 72 s).
- `qwen3.5:35b-a3b-coding-mxfp8` passed before (75 s, highest score of that job) → **failed now** with `port_never_opened`.
Overall pass rate dropped from 5/10 to 4/10 between the two jobs. For future runs, multiple samples per model would be cheaper than trusting a single data point.

**5. Fastest passing model (again): gemma4:e2b.**
15.7 s to a working app. Still the cleanest ratio of effort-to-result in the catalog.
